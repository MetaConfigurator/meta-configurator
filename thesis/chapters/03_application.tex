\chapter{Application}\label{sec:application}
%%% ===============================================================================

This chapter demonstrates the end-to-end application of the thesis workflow to \gls{mof} synthesis data. Structured experiment records are modeled in \gls{json}, transformed into semantically enriched \gls{rdf}/\gls{jsonld} using \gls{rml}, and then prepared for ontology-aware querying and graph-based analysis.

In line with the motivation in Chapters~\ref{sec:background} and~\ref{sec:design}, the main idea is to preserve laboratory protocol detail (materials, actions, and conditions) while adding formal semantics so that the same data can be interpreted consistently across tools and datasets.

\section{JSON Structure of the \glsentryshort{mof} Synthesis Dataset}\label{sec:application:metal:json-structure}

\gls{mof} synthesis refers to the stepwise preparation of crystalline coordination materials from metal precursors and organic linkers under controlled laboratory conditions. The synthesis process is not only defined by the final product, but also by the full protocol context (reagents, roles, action sequence, and process conditions), which should be represented in structured, machine-readable form. \cite{neubauer2026data}

Listings~\ref{lst:application_mof_json_example_s_1} and~\ref{lst:application_mof_json_example_s_2} show a representative subset of this dataset. Each top-level entry under \texttt{Synthesis[*]} encodes one complete run (e.g., \texttt{S-1}, \texttt{S-2}) with four core components: hardware setup, run metadata, ordered procedure phases (Prep/Reaction/Workup), and reagent inventory with functional roles.

\section{RML Mapping for Semantic Enrichment}\label{sec:application:metal:rml}

The \gls{rml} mappings in Listings~\ref{lst:application_mof_rml_example_1} and Listings~\ref{lst:application_mof_rml_example_2} show how the \gls{json} structures defined in Listings~\ref{lst:application_mof_json_example_s_1} and Listings~\ref{lst:application_mof_json_example_s_2} are transformed into typed \gls{rdf} resources, as described in Section~\ref{sec:background:rml}. In MetaConfigurator, this is the operational step that bridges conventional \gls{json} input and the \gls{rdf}/\gls{jsonld} authoring and query workflow.

\begin{lstlisting}[float=htbp,style=turtle,caption={Part of RML mapping focused on synthesis-level hardware and metadata},label={lst:application_mof_rml_example_1}]
@prefix rr: <http://www.w3.org/ns/r2rml#> .
@prefix rml: <http://semweb.mmlab.be/ns/rml#> .
@prefix ql: <http://semweb.mmlab.be/ns/ql#> .
@prefix ex: <http://example.org/> .
@prefix chmo: <http://purl.obolibrary.org/obo/CHMO_> .
@prefix obo: <http://purl.obolibrary.org/obo/> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .

<#SynthesisSource> rml:source "Data.json" ; rml:referenceFormulation ql:JSONPath ; rml:iterator "$.Synthesis[*]" .

<#SynthesisTriplesMap>
  rml:logicalSource <#SynthesisSource> ;
  rr:subjectMap [ rr:template "http://example.org/synthesis/{Hardware.Component[0]._id}" ; rr:class chmo:0000410 ] ;
  rr:predicateObjectMap [ rr:predicate ex:componentID ; rr:objectMap [ rml:reference "Hardware.Component[0]._id" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:componentType ; rr:objectMap [ rml:reference "Hardware.Component[0]._type" ] ] ;
  rr:predicateObjectMap [ rr:predicate rdfs:label ; rr:objectMap [ rml:reference "Metadata._description" ] ] ;
  rr:predicateObjectMap [ rr:predicate chmo:hasProduct ; rr:objectMap [ rml:reference "Metadata._product" ] ] .
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=turtle,caption={Part of RML mapping focused on reagents and preparation steps},label={lst:application_mof_rml_example_2}]
<#ReagentSource> rml:source "Data.json" ; rml:referenceFormulation ql:JSONPath ; rml:iterator "$.Synthesis[*].Reagents.Reagent[*]" .

<#ReagentTriplesMap>
  rml:logicalSource <#ReagentSource> ;
  rr:subjectMap [ rr:template "http://example.org/reagent/{_id}" ; rr:class obo:CHEBI_24431 ] ;
  rr:predicateObjectMap [ rr:predicate rdfs:label ; rr:objectMap [ rml:reference "_name" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:role ; rr:objectMap [ rml:reference "_role" ] ] .

<#PrepStepSource> rml:source "Data.json" ; rml:referenceFormulation ql:JSONPath ; rml:iterator "$.Synthesis[*].Procedure.Prep.Step[*]" .

<#PrepStepTriplesMap>
  rml:logicalSource <#PrepStepSource> ;
  rr:subjectMap [ rr:template "http://example.org/step/prep/{_vessel}-{_order}" ; rr:class obo:OBI_0000070 ] ;
  rr:predicateObjectMap [ rr:predicate ex:phase ; rr:objectMap [ rr:constant "Prep" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:action ; rr:objectMap [ rml:reference "$xml_type" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:amountValue ; rr:objectMap [ rml:reference "_amount.Value" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:amountUnit ; rr:objectMap [ rml:reference "_amount.Unit" ] ] .
\end{lstlisting}

\subsection{How Ontology Classes Add Semantics to JSON Data}\label{sec:application:metal:rml-class-semantics}

A central semantic effect of the mapping is ontology-based typing in \texttt{rr:class}. These class assignments transform plain JSON objects into explicit domain entities. For example, the mappings in Listings~\ref{lst:application_mof_rml_example_1} and~\ref{lst:application_mof_rml_example_2} assign the following classes:
\begin{itemize}
    \item \textbf{\texttt{chmo:0000410}} types each synthesis run as a chemistry synthesis entity.
    \item \textbf{\texttt{obo:CHEBI\_24431}} types each reagent node as a chemical entity.
    \item \textbf{\texttt{obo:OBI\_0000070}} types each procedure step as an experimental/process step entity.
\end{itemize}

This is important because \gls{json} keys such as \texttt{Synthesis}, \texttt{Reagent}, and \texttt{Step} are structural labels only. After \gls{rml} transformation, they become ontology-grounded \gls{rdf} nodes that can be queried by meaning and reused across datasets. This is the semantic lifting described in Sections~\ref{sec:background:jsonld} and~\ref{sec:background:rml}.
