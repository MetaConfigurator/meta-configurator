\chapter{Application}\label{sec:application}
%%% ===============================================================================

This chapter presents the end-to-end application of the thesis workflow to \gls{mof} synthesis data. Structured experiment records are first modeled in \gls{json}, then transformed into semantically enriched \gls{rdf}/\gls{jsonld} using \gls{rml}, and finally prepared for ontology-aware querying and graph-based analysis.

In line with the motivation in Chapters~\ref{sec:background} and~\ref{sec:design}, the main idea is to preserve laboratory protocol detail (materials, actions, and conditions) while adding formal semantics so that the same data can be interpreted consistently across tools and datasets \cite{neubauer2026data}.

\section{JSON Structure of the \glsentryshort{mof} Synthesis Dataset}\label{sec:application:metal:json-structure}

\gls{mof} synthesis refers to the stepwise preparation of crystalline coordination materials from metal precursors and organic linkers under controlled laboratory conditions. Beyond the final product, the process is characterized by full protocol context, including reagents, roles, action sequence, and process conditions, which should be represented in structured, machine-readable form \cite{neubauer2026data}.

Listings~\ref{lst:application_mof_json_example_s_1} and~\ref{lst:application_mof_json_example_s_2} show a representative subset of this dataset. Each top-level entry under \texttt{Synthesis[*]} encodes one complete run (e.g., \texttt{S-1}, \texttt{S-2}) with four core components: hardware setup, run metadata, ordered procedure phases (Prep/Reaction/Workup), and reagent inventory with functional roles.

\section{\glsentryshort{chmo} Ontology}\label{sec:application:metal:chmo-introduction}

The \gls{chmo} provides a controlled vocabulary for chemical methods and related experimental concepts, including data-acquisition techniques, preparation and separation procedures, and synthesis methods \cite{chmo_ontology}. Within the broader OBO ecosystem, \gls{chmo} is designed to complement other ontologies such as the \gls{obi}, which captures more general investigation processes and experimental context \cite{obi2016,jackson2021obofoundry}.

Using \gls{chmo} serves two purposes. First, it anchors synthesis resources to a domain-specific ontology rather than leaving them as generic data records. Second, it improves interoperability, because \gls{chmo}-typed entities can be combined with \gls{obi}- and \gls{chebi}-based descriptions in a shared \gls{rdf} graph.

\section{RML Mapping for CHMO Integration}\label{sec:application:metal:rml}

The \gls{rml} mappings in Listings~\ref{lst:application_mof_rml_example_1} and~\ref{lst:application_mof_rml_example_2} show how the \gls{json} structures defined in Listings~\ref{lst:application_mof_json_example_s_1} and~\ref{lst:application_mof_json_example_s_2} are transformed into typed \gls{rdf} resources, as described in Section~\ref{sec:background:rml}. In MetaConfigurator, this is the operational step that bridges conventional \gls{json} input and the \gls{rdf}/\gls{jsonld} authoring and query workflow.

Listings~\ref{lst:application_mof_rml_example_1} and~\ref{lst:application_mof_rml_example_2} demonstrate how the chemistry-oriented ontology design is operationalized as executable \gls{rml}. The first listing defines the synthesis-level mapping context: \texttt{rml:iterator "\$.Synthesis[*]"} selects each run in the input data, and \texttt{rr:subjectMap} creates one stable resource per run. By assigning \texttt{rr:class chmo:0000410}, each generated subject is explicitly typed as a \gls{chmo}-grounded synthesis entity rather than a generic node.

The same mapping block also shows how experimental metadata is semantically attached to the synthesis resource. Fields such as vessel identifier, vessel type, human-readable run label, and product are transferred through \texttt{rr:predicateObjectMap}. This mapping pattern preserves the operational laboratory context while simultaneously lifting the data into an ontology-aware representation that can be interpreted by semantic tools.

The second listing extends this principle to reagents and preparation steps. Reagent resources are typed with \texttt{obo:CHEBI\_24431}, aligning them with \gls{chebi}-based chemical entities, while preparation-step resources are typed with \texttt{obo:OBI\_0000070}, aligning procedural actions with \gls{obi}-based process semantics. As a result, the mapping does not merely copy JSON values into triples; it establishes a typed graph structure in which substances, methods, and process phases can be distinguished and queried systematically.

From a workflow perspective, this is crucial for the integration goals discussed in Chapters~\ref{sec:background} and~\ref{sec:design}. Once the mapped graph is loaded in MetaConfigurator, users can formulate \gls{sparql} queries over synthesis-level entities, reagent roles, and process steps without relying on fragile path-specific JSON logic. The same typed structure also improves graph visualization readability, because nodes and edges represent explicit domain semantics rather than implicit document structure.

\begin{lstlisting}[float=htbp,style=turtle,caption={Part of RML mapping focused on synthesis-level hardware and metadata},label={lst:application_mof_rml_example_1}]
@prefix rr: <http://www.w3.org/ns/r2rml#> .
@prefix rml: <http://semweb.mmlab.be/ns/rml#> .
@prefix ql: <http://semweb.mmlab.be/ns/ql#> .
@prefix ex: <http://example.org/> .
@prefix chmo: <http://purl.obolibrary.org/obo/CHMO_> .
@prefix obo: <http://purl.obolibrary.org/obo/> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .

<#SynthesisSource> rml:source "Data.json" ; rml:referenceFormulation ql:JSONPath ; rml:iterator "$.Synthesis[*]" .

<#SynthesisTriplesMap>
  rml:logicalSource <#SynthesisSource> ;
  rr:subjectMap [ rr:template "http://example.org/synthesis/{Hardware.Component[0]._id}" ; rr:class chmo:0000410 ] ;
  rr:predicateObjectMap [ rr:predicate ex:componentID ; rr:objectMap [ rml:reference "Hardware.Component[0]._id" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:componentType ; rr:objectMap [ rml:reference "Hardware.Component[0]._type" ] ] ;
  rr:predicateObjectMap [ rr:predicate rdfs:label ; rr:objectMap [ rml:reference "Metadata._description" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:hasProduct ; rr:objectMap [ rml:reference "Metadata._product" ] ] .
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=turtle,caption={Part of RML mapping focused on reagents and preparation steps},label={lst:application_mof_rml_example_2}]
<#ReagentSource> rml:source "Data.json" ; rml:referenceFormulation ql:JSONPath ; rml:iterator "$.Synthesis[*].Reagents.Reagent[*]" .

<#ReagentTriplesMap>
  rml:logicalSource <#ReagentSource> ;
  rr:subjectMap [ rr:template "http://example.org/reagent/{_id}" ; rr:class obo:CHEBI_24431 ] ;
  rr:predicateObjectMap [ rr:predicate rdfs:label ; rr:objectMap [ rml:reference "_name" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:role ; rr:objectMap [ rml:reference "_role" ] ] .

<#PrepStepSource> rml:source "Data.json" ; rml:referenceFormulation ql:JSONPath ; rml:iterator "$.Synthesis[*].Procedure.Prep.Step[*]" .

<#PrepStepTriplesMap>
  rml:logicalSource <#PrepStepSource> ;
  rr:subjectMap [ rr:template "http://example.org/step/prep/{_vessel}-{_order}" ; rr:class obo:OBI_0000070 ] ;
  rr:predicateObjectMap [ rr:predicate ex:phase ; rr:objectMap [ rr:constant "Prep" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:action ; rr:objectMap [ rml:reference "$xml_type" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:amountValue ; rr:objectMap [ rml:reference "_amount.Value" ] ] ;
  rr:predicateObjectMap [ rr:predicate ex:amountUnit ; rr:objectMap [ rml:reference "_amount.Unit" ] ] .
\end{lstlisting}

\section{AI-Assisted SPARQL Generation}

After applying the full mapping in Listing~\ref{lst:application_mof_rml_example_1} and Listing~\ref{lst:application_mof_rml_example_2}, users can query the resulting \gls{rdf} graph directly with \gls{sparql}. In practical laboratory workflows, however, writing correct \gls{sparql} often becomes a bottleneck, especially for users with strong domain expertise but limited query-language experience. Therefore, this chapter applies the same \gls{ai}-assisted interaction pattern introduced in Chapter~\ref{sec:design}: users formulate an intent in natural language, and a configured \gls{llm} proposes a first query draft.

For the \gls{mof} dataset, the main benefit is that users can start from experimental questions (e.g., ``Which Prep steps used which reagent, and in what amount?'') instead of manually constructing triple patterns from scratch. Listing~\ref{lst:application_mof_sparql_example_hint} shows such a concise user hint. Based on this hint and a representative subset of the current \gls{jsonld}/\gls{rdf} graph, the \gls{llm} generates a candidate query, shown in Listing~\ref{lst:application_mof_sparql_example}. Figure~\ref{fig:application:sparql_ai_generated_result} shows the result of executing this query, which retrieves all preparation steps with their associated reagents and amounts for each synthesis run.

This workflow lowers the entry barrier for \gls{sparql} authoring, reduces trial-and-error iterations in the query editor, and helps domain experts focus on scientific interpretation rather than query syntax details. At the same time, the generated query remains a draft artifact: it can contain modeling assumptions or predicate choices that require correction. Consequently, a user-controlled review step remains mandatory before execution.

\begin{lstlisting}[float=htbp,style=plaintext,caption={Example user hint for AI-assisted SPARQL generation}, label={lst:application_mof_sparql_example_hint}]
Retrieve all Prep steps with their reagents and amounts for each synthesis.
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=sparql,caption={AI-suggested SPARQL query for Prep steps with reagent and amount information}, label={lst:application_mof_sparql_example}]
PREFIX ex: <http://example.org/>
PREFIX chmo: <http://purl.obolibrary.org/obo/CHMO_>
PREFIX obo: <http://purl.obolibrary.org/obo/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?synthesis ?step ?reagentLabel ?amount ?unit
WHERE {
  ?synthesis obo:BFO_0000051 ?step .
  ?step ex:phase "Prep" ;
        obo:RO_0000057 ?reagent ;
        ex:amountValue ?amount ;
        ex:amountUnit ?unit .
  ?reagent rdfs:label ?reagentLabel .
}
ORDER BY ?synthesis ?step
\end{lstlisting}

\section{\glsentryshort{kg} Visualization and Editing}\label{sec:application:knowledge-graph-visualization}

After conversion of the synthesis records to \gls{jsonld}, graph visualization becomes a direct, low-friction step in the workflow: users can open the Visualization view and inspect the same underlying \gls{rdf} data as a \gls{kg}, without writing additional transformation code. In this representation, synthesis runs, reagents, and process steps are shown as connected nodes, while semantic relations appear as labeled edges. Figure~\ref{fig:application:kg_visualization} illustrates this view for the mapped \gls{mof} data.

This immediate \gls{jsonld}-to-\gls{kg} transition improves usability for non-\gls{sparql} users. Instead of interpreting nested JSON structures or triple tables, users can understand protocol structure visually: which reagent belongs to which synthesis, how Prep/Reaction/Workup steps are linked, and where key metadata is attached. As a result, semantic consistency checks can be performed faster during exploratory analysis.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=.99\linewidth]{../figures/MC-RDF-KG-2.png}
  \caption{Visualization of the JSON-LD data in Listing~\ref{lst:application_mof_jsonld_example_s_1}, rendered in the Visualization dialog.}
  \label{fig:application:kg_visualization}
\end{figure}

The same interface also supports lightweight editing of graph content. When a node is selected, the properties panel exposes its current predicates and values and allows users to add, edit, or delete properties directly. Node-level operations such as rename, add, and delete are also available. This means users can refine both structure and annotations in place, rather than switching between multiple tools.

Crucially, these edits are synchronized with the underlying \gls{jsonld}/\gls{rdf} model. In other words, the \gls{kg} is not only a static visualization layer; it is an interactive authoring surface. For end users, this reduces modeling effort and lowers the barrier to maintaining high-quality semantic data, because corrections can be performed where relationships are most intuitive: directly in the graph.

In addition, users can filter the graph semantically with \gls{sparql} and visualize only the query result as a subgraph. This is particularly useful when the full dataset is large: instead of rendering all nodes, users can formulate a focused query (e.g., only Prep-related resources or only selected reagents), execute it, and inspect the returned structure directly in graph form. Hence, \gls{sparql}-based filtering and \gls{kg} visualization work together as a combined analysis workflow.
