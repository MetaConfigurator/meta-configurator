\section{RDF Authoring Panel}\label{sec:design:rdf-authoring-panel}

The \gls{rdf} authoring functionality is implemented as a dedicated, separate panel in MetaConfigurator, located in \texttt{meta\_configurator/src/components/panels/rdf}. This design keeps semantic authoring concerns modular while still integrating with the existing panel configurations, path selection, and data editing flow of MetaConfigurator \cite{metaconfigurator2024}. At runtime, the entry component \texttt{RdfPanel.vue} validates whether the current data can be treated as \gls{jsonld} and provides immediate warnings for missing \texttt{@context}, invalid syntax, and non-\gls{jsonld} input.

Figure \ref{fig:rdf_authoring_panel:json_data} shows a snapshot of the \gls{rdf} panel in action, loaded with the \gls{json} example from \ref{lst:background_json_example}. Since the data is provided in \gls{json} format, the panel indicates that it must first be converted to \gls{jsonld} before further processing. The user can proceed in two ways: either by directly supplying a \gls{jsonld} document in the Text View, or by using the \textbf{JSON-to-JSON-LD} conversion tool discussed in Section \ref{sec:design:rdf-authoring-panel:rml-mapping-tool}.

\subsection{RML Mapping Tool}
\label{sec:design:rdf-authoring-panel:rml-mapping-tool}

If the input data is not already expressed as \gls{jsonld}, the \gls{rdf} panel provides a conversion tool that enables users to transform \gls{json} data into \gls{rdf}-compatible \gls{jsonld} using \gls{rml} mappings. \gls{rml} extends the \gls{r2rml} mapping language and allows the transformation of heterogeneous data sources such as \gls{json}, \gls{csv}, or \gls{xml} into \gls{rdf} graphs \cite{dimou2014rml}, as discussed in Section~\ref{sec:background:rml}. Figure~\ref{fig:rdf_authoring_panel:rml_1} illustrates the dialog used to define such mappings.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=.75\linewidth]{../figures/MC-RDF-RML-1.png}
  \caption{The RML mapping dialog for JSON-to-JSON-LD conversion.}
  \label{fig:rdf_authoring_panel:rml_1}
\end{figure}

The dialog allows users to either provide an existing \gls{rml} mapping or generate one automatically with \gls{ai} assistance. In the latter case, the user supplies mapping hints that describe how elements of the input \gls{json} data should correspond to \gls{rdf} properties or classes. These hints, together with a representative subset of the input data, are sent to an external \gls{ai} API to generate a candidate mapping.

\begin{lstlisting}[float=htbp,style=plaintext,caption={Sample RML hints which describe the mapping from JSON in \ref{lst:background_json_example} to a minimal JSON-LD in \ref{lst:background_jsonld_example_simple}}, label={lst:metaconfigurator_interface_rdf:rml_hint_simple}]
Convert my JSON Data to JSON-LD such that I have two nodes, each node representing a parameter or metric.
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=turtle, caption={RML mapping which converts JSON data in Listing~\ref{lst:background_json_example} to JSON-LD in Listing~\ref{lst:background_jsonld_example_simple}}, label={lst:background_rml_mapping_simple}]
@prefix rr: <http://www.w3.org/ns/r2rml#> .
@prefix rml: <http://semweb.mmlab.be/ns/rml#> .
@prefix ql: <http://semweb.mmlab.be/ns/ql#> .
@prefix ex: <http://example.com/ns#> .

<#ParameterMapping>
  rml:logicalSource [
    rml:source "Data.json" ;
    rml:referenceFormulation ql:JSONPath ;
    rml:iterator "$.parameters[*]"
  ] ;

  rr:subjectMap [
    rr:template "http://example.com/parameter/{name}" ;
    rr:class ex:Parameter
  ] ;

  rr:predicateObjectMap [
    rr:predicate ex:parameterName ;
    rr:objectMap [ rml:reference "name" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:parameterValue ;
    rr:objectMap [ rml:reference "value" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:parameterUnit ;
    rr:objectMap [ rml:reference "unit" ]
  ] .

<#MetricMapping>
  rml:logicalSource [
    rml:source "Data.json" ;
    rml:referenceFormulation ql:JSONPath ;
    rml:iterator "$.metrics[*]"
  ] ;

  rr:subjectMap [
    rr:template "http://example.com/metric/{name}" ;
    rr:class ex:Metric
  ] ;

  rr:predicateObjectMap [
    rr:predicate ex:metricName ;
    rr:objectMap [ rml:reference "name" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:metricValue ;
    rr:objectMap [ rml:reference "value" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:metricUnit ;
    rr:objectMap [ rml:reference "unit" ]
  ] .
\end{lstlisting}

Internally, the system constructs a structured prompt that includes example mappings and fragments of the input data. This prompt guides the language model to produce a valid \gls{rml} mapping in Turtle syntax that conforms to the target \gls{jsonld} structure. The prompt constrains the model to return only the mapping definition while adhering to \gls{rml} rules, thereby improving reliability and syntactic correctness.

The mapping editor is implemented using the CodeMirror component, a widely used web-based code editor that provides syntax highlighting and automatic indentation \cite{codemirror}. Mapping validity is checked by the mapping service through debounced live validation. This allows users to inspect, refine, and validate the generated mapping directly within the interface before applying it to transform the \gls{json} input into \gls{jsonld}.

Users may either provide a \gls{rml} mapping in Turtle syntax directly or use the \gls{ai}-assisted generation feature by supplying mapping hints. The resulting mapping is displayed in the CodeMirror editor, where it can be reviewed and edited as needed. Since the mapping is generated by \gls{ai}, user verification and refinement are recommended. Once finalized, the mapping can be applied to convert the input \gls{json} data into \gls{jsonld}, which can then be further managed in the \gls{rdf} authoring panel.

The examples in this section show that even a minimal hint can already produce a correct mapping for simple targets. In particular, the simple hint in Listing \ref{lst:metaconfigurator_interface_rdf:rml_hint_simple} can still lead to a valid \gls{rml} mapping (Listing \ref{lst:background_rml_mapping_simple}) and a corresponding \gls{jsonld} output (Listing \ref{lst:background_jsonld_example_simple}).

\begin{lstlisting}[float=htbp,style=json, caption={JSON-LD representation of the simulation result, obtained after applying the minimal RML mapping hint in Listing~\ref{lst:metaconfigurator_interface_rdf:rml_hint_simple}}, label={lst:background_jsonld_example_simple}]

    "ex": "http://example.com/ns#"
  },

    {
      "@id": "http://example.com/metric/max_von_mises_stress",

      "ex:metricName": "max_von_mises_stress",
      "ex:metricUnit": "MPa",

        "@type": "http://www.w3.org/2001/XMLSchema#double",

      }
    },
    {
      "@id": "http://example.com/parameter/element_size",

Therefore, detailed hints such as Listing \ref{lst:metaconfigurator_interface_rdf:rml_hint_complex} are needed when the target output must follow specific vocabularies, resource patterns, and links between entities. With this level of instruction, the generated mapping can match the expected structure in Listing \ref{lst:background_rml_mapping} and produce the intended \gls{jsonld} in Listing \ref{lst:background_jsonld_example}.

\subsection{Panel Composition and UI Structure}
\label{sec:design:rdf-authoring-panel:panel-composition}

The core editing surface is implemented in \texttt{RdfEditorPanel.vue} as a PrimeVue Tabs\footnote{\url{https://primevue.org/tabs/}} container. The implementation is primarily organized into two tabs:
\begin{itemize}
    \item \textbf{Context tab}: for editing \texttt{@context} definitions.
    \item \textbf{Triples tab}: for browsing and editing \gls{rdf} triples.
\end{itemize}

This explicit split mirrors the conceptual separation in \gls{jsonld} between vocabulary declarations and graph assertions \cite{jsonld2014}. The panel therefore supports both semantic modeling tasks (namespace/prefix handling) and graph authoring tasks (subject-predicate-object manipulation) in one workflow.

When the current document is invalid or not in the expected \gls{jsonld} form, both tabs become visually disabled. This prevents accidental modifications when parsing preconditions are not satisfied.

\subsection{Context Tab Implementation}
\label{sec:design:rdf-authoring-panel:context-tab-implementation}

The Context tab is implemented by \texttt{RdfContextEditorPanel.vue}. It reuses MetaConfigurator's \gls{gui} editor and applies a predefined JSON Schema so that \texttt{@context} can be edited through structured forms rather than plain text only. The selection updates are synchronized with the text editor view, enabling users to navigate to and manipulate context entries consistently with other panels.

\subsection{Triples Tab Implementation}
\label{sec:design:rdf-authoring-panel:triples-tab-implementation}

The Triples tab is implemented by \texttt{RdfTripleEditorPanel.vue}. It shows triples in a PrimeVue \texttt{DataTable} \footnote{\url{https://primevue.org/datatable/}} with:
\begin{itemize}
    \item paging, sorting, and column/global filtering,
    \item single-row selection synchronized with document path selection in the text editor,
    \item actions for add, edit, delete, export, \gls{sparql}, and visualization.
\end{itemize}

Triple editing is handled through \texttt{TripleDetailsDialog.vue}, which allows controlled entry of subject, predicate, and object terms, including typed literals with XML Schema datatypes. Changes are executed through \texttt{TripleEditorService} and propagated to the central \gls{rdf} store. Figure \ref{{fig:rdf_authoring_panel:triple-modal}} shows the edit modal for triple editing.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=.85\linewidth]{../figures/MC-RDF-Modal.png}
  \caption{Edit modal for triple editing.}
  \label{fig:rdf_authoring_panel:triple-modal}
\end{figure}

Data changes occur from two sources: the Text View (raw JSON-LD data) or the RDF Panel. When a change is made in the Text View, \texttt{rdfStoreManager.ts} rebuilds the \gls{rdf} graph and updates the RDF Panel (Figure \ref{fig:rdf_authoring_panel:change_propagation_json_editor}).

Conversely, when a triple is edited, removed, or added in Triples tab, \texttt{TripleEditorService} validates and applies the change through \texttt{rdfStoreManager.ts}. If successful, \texttt{jsonLdNodeManager.ts} rebuilds \gls{jsonld} from the updated store and writes it back to the Data Editor state, which updates the Text View accordingly (Figure \ref{fig:rdf_authoring_panel:change_propagation_rdf_view}).
Change in the Context tab is propagated in the same way as changes to the GUI View, since underlying data is still treated as JSON data and processed by the same synchronization workflow.

\subsection{RDF Store and Data Synchronization}
\label{sec:design:rdf-authoring-panel:rdf-store-and-sync}

The internal semantic state is managed by \texttt{rdfStoreManager.ts}, which uses RDFLib to parse and serialize data \cite{rdflib_docs}. On each source-data update, the manager:
\begin{enumerate}
    \item resolves \texttt{@context} data into namespace mappings,
    \item parses \gls{jsonld} into an indexed \gls{rdf} graph and derives a reactive list of Statements,
    \item exposes reactive Statements, warnings, and errors to UI components.
\end{enumerate}

\begin{definition}[RDFLib Statement]
In this implementation, an RDFLib Statement is the runtime representation of one \gls{rdf} triple of the form $(s, p, o)$, where $s$ (subject), $p$ (predicate), and $o$ (object) are \gls{rdf} terms. In \texttt{rdflib.js}, these terms are represented as \texttt{NamedNode}, \texttt{BlankNode}, and \texttt{Literal} objects, and each triple is stored as a Statement. A collection of such Statements forms the in-memory graph used by the panel \cite{rdflib_docs}.

\end{definition}

This definition is directly reflected in the implementation workflow: the internal store parses \gls{jsonld} into a reactive list of Statement objects, the triples table renders this list, and editing actions create or modify individual statements for add/edit/delete operations. Statement-level matching (subject, predicate, object) is used to synchronize table selection with text-editor paths.

To keep the \gls{json} editor and \gls{rdf} authoring panel consistent, the implementation maps \gls{rdf} statements back into \gls{jsonld} data and computes path correspondences between selected triples and document positions. This enables bidirectional synchronization: selecting a triple can focus the matching JSON path, and raw \gls{json} data changes trigger store rebuilds.

\subsection{Synchronization Between Text View and RDF View}
\label{sec:design:rdf-authoring-panel:view-synchronization}

Synchronization between the Text View (\gls{jsonld} editor) and the \gls{rdf} View (triples DataTable) is implemented as two directional mapping procedures that operate on \texttt{\$rdf.Statement} matching.

\subsubsection{Text View to RDF View}
\label{sec:design:rdf-authoring-panel:view-synchronization:text-to-rdf}

When the current selection changes in the Text View, the selected JSON path is read from the editor state and used to derive a focused semantic fragment.

This extraction step constructs a minimal \gls{jsonld} fragment that preserves the original \texttt{@context} and contains exactly one node in \texttt{@graph}. The node contains:
\begin{itemize}
    \item the subject identifier (\texttt{@id}),
    \item the selected predicate,
    \item the selected object value (including array-element handling when needed).
\end{itemize}

The minimal fragment is then parsed by RDFLib into a temporary graph. If the parse yields exactly one statement, this statement is compared against the current statement list using subject, predicate, and object equality. The resulting index is used to focus and select the matching row in the DataTable.

\subsubsection{RDF View to Text View}
\label{sec:design:rdf-authoring-panel:view-synchronization:rdf-to-text}

When a row is selected (or clicked) in the triples DataTable, the selected statement's subject, predicate, and object are used to search the corresponding location in the \gls{jsonld} document. Path matching is performed in three stages:
\begin{enumerate}
    \item find the candidate node by matching \texttt{@id} with the subject,
    \item inside that node, find the matching predicate key,
    \item inside that predicate value, match the object (as \texttt{@id}, \texttt{@value}, or scalar value).
\end{enumerate}

To improve robustness, matching considers semantic term equivalence and checks both compact prefixed terms and full \glspl{iri}. Therefore, values such as \texttt{schema:value} and \texttt{https://schema.org/value} are treated as equivalent during lookup. Once the path is found, the editor view navigates to the corresponding JSON location.

\subsection{Integrated Extensions in the Triple Workflow}
\label{sec:design:rdf-authoring-panel:integrated-extensions}

The Triple tab also integrates two advanced features:
\begin{itemize}
    \item \textbf{SPARQL dialog} (\texttt{SparqlEditor.vue}): query execution and result inspection aligned with SPARQL standards \cite{sparql_standard}.
    \item \textbf{Graph visualization} (\texttt{RdfVisualizer.vue}): interactive inspection of currently filtered triples as a graph representation.
\end{itemize}

Together, this implementation turns the RDF panel into a full authoring environment: users can define context terms, curate triples, run semantic queries, and visually inspect graph structures in a single, cohesive interface.



\section{Query with SPARQL}\label{sec:design:query-with-sparql}
The \gls{sparql} query functionality is implemented in \texttt{SparqlEditor.vue} as a dialog with three tabs: \textbf{Query View}, \textbf{Result View}, and \textbf{Visualization}. During implementation, RDFLib query capabilities were evaluated first, but they proved limited for the required in-browser query workflow and did not provide full support for SPARQL~1.2. Query execution is therefore performed in-browser using Comunica's \texttt{QueryEngine} \cite{comunica_query_sparql}. According to Comunica's official documentation, the engine supports SPARQL 1.2 query-related specifications \cite{comunica_supported_specs}.

In the \textbf{Query View}, users can write and validate queries in a CodeMirror editor and execute them against the current in-memory \gls{rdf} graph. Query syntax is checked through debounced live validation using the SparqlJS parser, which provides immediate feedback while editing \cite{codemirror,sparqljs}. The same view also provides \gls{ai}-assisted \gls{sparql} generation. Similar to the \gls{ai}-assisted \gls{rml} workflow described in Section~\ref{sec:design:rdf-authoring-panel:rml-mapping-tool}, user hints are combined with prompt-engineering instructions and a representative subset of the current \gls{jsonld} data, then sent to the configured \gls{llm} to generate a draft \gls{sparql} query.

\subsection{Showcase: AI-Assisted SPARQL Query Generation}
\label{sec:design:query-with-sparql:ai-showcase}

To demonstrate the \gls{ai}-assisted workflow, we assume that the simulation \gls{json} from Listing~\ref{lst:background_json_example} has been extended with additional \texttt{element\_size} and \texttt{max\_von\_mises\_stress} value pairs (e.g., from multiple simulation runs), and then transformed into \gls{jsonld} using the mapping workflow described earlier. In this extended dataset, metric measurements are available for multiple element sizes, for example \texttt{0.003125}, \texttt{0.00625}, \texttt{0.0125}, \texttt{0.05} and \texttt{0.1}.

In this scenario, the user provides a natural-language hint in the Query View to request a query that returns paired values of mesh element size and maximum von Mises stress. An example hint is shown in Listing~\ref{lst:design_sparql_ai_hint}. The hint and a subset of the current \gls{jsonld} graph are sent to the configured \gls{llm}, which proposes a syntactically valid draft query that can then be reviewed and refined by the user. A representative generated query is shown in Listing~\ref{lst:design_sparql_ai_generated}, and its integration in the dialog is illustrated in Figure~\ref{fig:rdf_authoring_panel:sparql_ai_generated}.

\begin{lstlisting}[float=htbp,style=plaintext,caption={Example user hint for AI-assisted SPARQL generation}, label={lst:design_sparql_ai_hint}]
Return pairs of element_size and max_von_mises_stress.
For each run, show the numeric element size value and the numeric stress value.
Sort by element_size ascending.
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=sparql,caption={AI-suggested SPARQL query for \texttt{element\_size} / \texttt{max\_von\_mises\_stress} pairs}, label={lst:design_sparql_ai_generated}]
PREFIX m4i: <http://w3id.org/nfdi4ing/metadata4ing#>
PREFIX schema: <https://schema.org/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX unit: <http://qudt.org/vocab/unit/>
PREFIX local: <https://www.example.org/>

SELECT ?element_size ?max_von_mises_stress WHERE {
  ?method a m4i:Method .
  ?method m4i:hasParameter ?element_size_param .
  ?element_size_param a schema:PropertyValue ;
                      schema:value ?element_size .
  ?method m4i:investigates ?stress_param .
  ?stress_param a schema:PropertyValue ;
                schema:value ?max_von_mises_stress .
} ORDER BY ASC(?element_size)
\end{lstlisting}

This showcase illustrates the intended interaction pattern: \gls{ai} is used to accelerate query authoring, while final semantic and syntactic validation remains under user control before execution. Human-in-the-loop review is therefore essential in both \gls{ai}-generated \gls{rml} mappings and \gls{ai}-generated \gls{sparql} queries, and users can inspect and refine both results before applying or executing them.

The dialog supports two execution modes controlled by a toggle: \textbf{Visualization Off} and \textbf{Visualization On}. With \textbf{Visualization Off}, standard SELECT-style querying is used and results are shown in the \textbf{Result View}. There, the PrimeVue DataTable columns are generated dynamically from the returned variables, and export is provided as CSV. With \textbf{Visualization On}, the query mode switches to CONSTRUCT-based graph output; the same result table is still populated, but export options target RDF serializations (e.g., Turtle, N-Triples, RDF/XML) for graph-oriented workflows.

\section{Knowledge Graph Visualization}\label{sec:design:knowledge-graph-visualization}
