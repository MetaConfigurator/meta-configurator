\section{RDF Authoring Panel}\label{sec:design:rdf-authoring-panel}

The \gls{rdf} authoring functionality is implemented as a dedicated, separate panel in MetaConfigurator, located in \texttt{meta\_configurator/src/components/panels/rdf}. This design keeps semantic authoring concerns modular while still integrating with the existing panel configurations, path selection, and data editing flow of MetaConfigurator \cite{metaconfigurator2024}. At runtime, the entry component \texttt{RdfPanel.vue} validates whether the current data can be treated as \gls{jsonld} and provides immediate warnings for missing \texttt{@context}, invalid syntax, and non-\gls{jsonld} input.

Figure~\ref{fig:rdf_authoring_panel:json_data} shows a snapshot of the \gls{rdf} panel in action, loaded with the \gls{json} example from~\ref{lst:background_json_example}. Since the data is provided in \gls{json} format, the panel indicates that it must first be converted to \gls{jsonld} before further processing. The user can proceed in two ways: either by directly supplying a \gls{jsonld} document in the Text View, or by using the JSON-to-JSON-LD conversion tool discussed in Section~\ref{sec:design:rdf-authoring-panel:rml-mapping-tool}.

\subsection{RML Mapping Tool}
\label{sec:design:rdf-authoring-panel:rml-mapping-tool}

If the input data is not already expressed as \gls{jsonld}, the \gls{rdf} panel provides a conversion tool that enables users to transform \gls{json} data into \gls{rdf}-compatible \gls{jsonld} using \gls{rml} mappings. \gls{rml} extends the \gls{r2rml} mapping language and allows the transformation of heterogeneous data sources such as \gls{json}, \gls{csv}, or \gls{xml} into \gls{rdf} graphs \cite{dimou2014rml}, as discussed in Section~\ref{sec:background:rml}. Figure~\ref{fig:rdf_authoring_panel:rml_1} illustrates the dialog used to define such mappings.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=.75\linewidth]{../figures/MC-RDF-RML-1.png}
  \caption{The RML mapping dialog for JSON-to-JSON-LD conversion.}
  \label{fig:rdf_authoring_panel:rml_1}
\end{figure}

The dialog allows users to either provide an existing \gls{rml} mapping or generate one automatically with \gls{ai} assistance. In the latter case, the user supplies mapping hints that describe how elements of the input \gls{json} data should be mapped to \gls{rdf} properties or classes. These hints, together with a representative subset of the input data, are sent to an external \gls{ai} API to generate a candidate mapping.

\begin{lstlisting}[float=htbp,style=plaintext,caption={Sample RML hints which describe the mapping from JSON in~\ref{lst:background_json_example} to a minimal JSON-LD in~\ref{lst:background_jsonld_example_simple}}, label={lst:metaconfigurator_interface_rdf:rml_hint_simple}]
Convert my JSON Data to JSON-LD such that I have two nodes, each node representing a parameter or metric.
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=turtle, caption={RML mapping which converts JSON data in Listing~\ref{lst:background_json_example} to JSON-LD in Listing~\ref{lst:background_jsonld_example_simple}}, label={lst:background_rml_mapping_simple}]
@prefix rr: <http://www.w3.org/ns/r2rml#> .
@prefix rml: <http://semweb.mmlab.be/ns/rml#> .
@prefix ql: <http://semweb.mmlab.be/ns/ql#> .
@prefix ex: <http://example.com/ns#> .

<#ParameterMapping>
  rml:logicalSource [
    rml:source "Data.json" ;
    rml:referenceFormulation ql:JSONPath ;
    rml:iterator "$.parameters[*]"
  ] ;

  rr:subjectMap [
    rr:template "http://example.com/parameter/{name}" ;
    rr:class ex:Parameter
  ] ;

  rr:predicateObjectMap [
    rr:predicate ex:parameterName ;
    rr:objectMap [ rml:reference "name" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:parameterValue ;
    rr:objectMap [ rml:reference "value" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:parameterUnit ;
    rr:objectMap [ rml:reference "unit" ]
  ] .

<#MetricMapping>
  rml:logicalSource [
    rml:source "Data.json" ;
    rml:referenceFormulation ql:JSONPath ;
    rml:iterator "$.metrics[*]"
  ] ;

  rr:subjectMap [
    rr:template "http://example.com/metric/{name}" ;
    rr:class ex:Metric
  ] ;

  rr:predicateObjectMap [
    rr:predicate ex:metricName ;
    rr:objectMap [ rml:reference "name" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:metricValue ;
    rr:objectMap [ rml:reference "value" ]
  ] ;
  rr:predicateObjectMap [
    rr:predicate ex:metricUnit ;
    rr:objectMap [ rml:reference "unit" ]
  ] .
\end{lstlisting}

Internally, the system constructs a structured prompt that includes example mappings and fragments of the input data. This prompt guides the language model to produce a valid \gls{rml} mapping in Turtle syntax that conforms to the target \gls{jsonld} structure. The prompt constrains the model to return only the mapping definition while adhering to \gls{rml} rules, thereby improving reliability and syntactic correctness. In simplified form, the prompt contains: (i) role/instruction constraints, (ii) required prefixes and output format, (iii) representative input-data fragments, and (iv) user-provided mapping hints.

The mapping editor is implemented using the CodeMirror component, a widely used web-based code editor that provides syntax highlighting and automatic indentation \cite{codemirror}. Mapping validity is checked by the mapping service through debounced live validation. This allows users to inspect, refine, and validate the generated mapping directly within the interface before applying it to transform the \gls{json} input into \gls{jsonld}.

Users may either provide a \gls{rml} mapping in Turtle syntax directly or use the \gls{ai}-assisted generation feature by supplying mapping hints. The resulting mapping is displayed in the CodeMirror editor, where it can be reviewed and edited as needed. Because \gls{ai}-generated mappings may contain semantic or syntactic errors, users must verify them before execution. Once finalized, the mapping can be applied to convert the input \gls{json} data into \gls{jsonld}, which can then be further managed in the \gls{rdf} authoring panel.

The examples in this section show that even a minimal hint can already produce a correct mapping for simple targets. In particular, the short natural-language hint shown above can still lead to a valid \gls{rml} mapping (Listing~\ref{lst:background_rml_mapping_simple}) and a corresponding \gls{jsonld} output (Listing~\ref{lst:background_jsonld_example_simple}).

\begin{lstlisting}[float=htbp,style=json, caption={JSON-LD representation of the simulation result, obtained after applying the minimal RML mapping hint shown above}, label={lst:background_jsonld_example_simple}]
{
  "@context": {
    "ex": "http://example.com/ns#"
  },
  "@graph": [
    {
      "@id": "http://example.com/metric/max_von_mises_stress",
      "@type": "ex:Metric",
      "ex:metricName": "max_von_mises_stress",
      "ex:metricUnit": "MPa",
      "ex:metricValue": {
        "@type": "http://www.w3.org/2001/XMLSchema#double",
        "@value": "2.997915075586333E8"
      }
    },
    {
      "@id": "http://example.com/parameter/element_size",
      "@type": "ex:Parameter",
      "ex:parameterName": "element_size",
      "ex:parameterUnit": "m",
      "ex:parameterValue": {
        "@type": "http://www.w3.org/2001/XMLSchema#double",
        "@value": "2.5E-2"
      }
    }
  ]
}  
\end{lstlisting}

However, realistic \gls{rml} authoring is often more complex and usually requires precise guidance to obtain the intended semantic structure. Therefore, detailed hints such as Listing~\ref{lst:metaconfigurator_interface_rdf:rml_hint_complex} are needed when the target output must follow specific vocabularies, resource patterns, and links between entities. With this level of instruction, the generated mapping can match the expected structure in Listing~\ref{lst:background_rml_mapping} and produce the intended \gls{jsonld} in Listing~\ref{lst:background_jsonld_example}. 

\begin{lstlisting}[float=htbp,style=plaintext,basicstyle=\ttfamily\fontsize{7}{8}\selectfont\color{black},caption={Sample RML hints which describe the mapping from JSON in~\ref{lst:background_json_example} to JSON-LD in~\ref{lst:background_jsonld_example}}, label={lst:metaconfigurator_interface_rdf:rml_hint_complex}]

m4i: http://w3id.org/nfdi4ing/metadata4ing#

local: https://www.example.org/

\subsection{Panel Composition and UI Structure}
\label{sec:design:rdf-authoring-panel:panel-composition}

The core editing surface is implemented in \texttt{RdfEditorPanel.vue} as a PrimeVue Tabs\footnote{\url{https://primevue.org/tabs/}} container. The implementation is primarily organized into two tabs:
\begin{itemize}
    \item \textbf{Context tab}: for editing \texttt{@context} definitions.
    \item \textbf{Triples tab}: for browsing and editing \gls{rdf} triples.
\end{itemize}

This explicit split mirrors the conceptual separation in \gls{jsonld} between vocabulary declarations and graph assertions \cite{jsonld2014}, and therefore supports both semantic modeling tasks (namespace/prefix handling) and graph authoring tasks (subject-predicate-object manipulation) in one workflow.

When the current document is invalid or not in the expected \gls{jsonld} form, both tabs become visually disabled. This prevents accidental modifications when parsing preconditions are not satisfied.

\subsection{Context Tab Implementation}
\label{sec:design:rdf-authoring-panel:context-tab-implementation}

The Context tab is implemented by \texttt{RdfContextEditorPanel.vue}. It reuses MetaConfigurator's \gls{gui} editor and applies a predefined JSON Schema so that \texttt{@context} can be edited through structured forms rather than plain text only. The selection updates are synchronized with the text editor view, enabling users to navigate to and manipulate context entries consistently with other panels.

\subsection{Triples Tab Implementation}
\label{sec:design:rdf-authoring-panel:triples-tab-implementation}

The Triples tab is implemented by \texttt{RdfTripleEditorPanel.vue}. It shows triples in a PrimeVue \texttt{DataTable} \footnote{\url{https://primevue.org/datatable/}} with:
\begin{itemize}
    \item paging, sorting, and column/global filtering,
    \item single-row selection synchronized with document path selection in the text editor,
    \item actions for add, edit, delete, export, \gls{sparql}, and visualization.
\end{itemize}

Triple editing is handled through \texttt{TripleDetailsDialog.vue}, which allows controlled entry of subject, predicate, and object terms, including typed literals with XML Schema datatypes. Changes are executed through \texttt{TripleEditorService} and propagated to the central \gls{rdf} store. Figure~\ref{fig:rdf_authoring_panel:triple_modal} shows the edit modal for triple editing.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=.85\linewidth]{../figures/MC-RDF-Modal.png}
  \caption{Edit modal for triple editing.}
  \label{fig:rdf_authoring_panel:triple_modal}
\end{figure}

Data changes occur from two sources: the Text View (raw JSON-LD data) or the RDF Panel. When a change is made in the Text View, \texttt{rdfStoreManager.ts} rebuilds the \gls{rdf} graph and updates the RDF Panel (Figure~\ref{fig:rdf_authoring_panel:change_propagation_json_editor}).

Conversely, when a triple is edited, removed, or added in Triples tab, \texttt{TripleEditorService} validates and applies the change through \texttt{rdfStoreManager.ts}. If successful, \texttt{jsonLdNodeManager.ts} rebuilds \gls{jsonld} from the updated store and writes it back to the Data Editor state, which updates the Text View accordingly (Figure~\ref{fig:rdf_authoring_panel:change_propagation_rdf_view}).
Change in the Context tab is propagated in the same way as changes to the GUI View, since underlying data is still treated as JSON data and processed by the same synchronization workflow.

\subsection{RDF Store and Data Synchronization}
\label{sec:design:rdf-authoring-panel:rdf-store-and-sync}

The internal semantic state is managed by \texttt{rdfStoreManager.ts}, which uses \texttt{rdflib.js} to parse and serialize data \cite{rdflib_docs}. \texttt{rdflib.js} was selected because it provides mature browser-side handling of \gls{jsonld}/\gls{rdf} terms and integrates directly with statement-level editing operations used by the panel. On each source-data update, the manager:
\begin{enumerate}
    \item resolves \texttt{@context} data into namespace mappings,
    \item parses \gls{jsonld} into an indexed \gls{rdf} graph and derives a reactive list of Statements,
    \item exposes reactive Statements, warnings, and errors to UI components.
\end{enumerate}

\begin{definition}[RDFLib Statement]
In this implementation, an RDFLib Statement is the runtime representation of one \gls{rdf} triple of the form $(s, p, o)$, where $s$ (subject), $p$ (predicate), and $o$ (object) are \gls{rdf} terms. In \texttt{rdflib.js}, these terms are represented as \texttt{NamedNode}, \texttt{BlankNode}, and \texttt{Literal} objects, and each triple is stored as a Statement. Briefly, a \texttt{NamedNode} represents an identified resource via an IRI (e.g., ontology classes, predicates, or linked entities), a \texttt{BlankNode} represents an unnamed local resource without a global IRI, and a \texttt{Literal} represents a concrete value such as text, number, or date (optionally with datatype or language tag). A collection of such Statements forms the in-memory graph used by the panel \cite{rdflib_docs}.

\end{definition}

This definition is directly reflected in the implementation workflow: the internal store parses \gls{jsonld} into a reactive list of Statement objects, the triples table renders this list, and editing actions create or modify individual statements for add/edit/delete operations. Statement-level matching (subject, predicate, object) is used to synchronize table selection with text-editor paths.

To keep the \gls{json} editor and \gls{rdf} authoring panel consistent, the implementation maps \gls{rdf} statements back into \gls{jsonld} data and computes path correspondences between selected triples and document positions. This enables bidirectional synchronization: selecting a triple can focus the matching JSON path, and raw \gls{json} data changes trigger store rebuilds.

\subsection{Linked Selection Between Text View and RDF View}
\label{sec:design:rdf-authoring-panel:linked-selection}

Linked selection between the Text View and the \gls{rdf} View enables coordinated highlighting of corresponding elements across both representations. When a user selects a section of the JSON-LD document, the corresponding RDF statement is identified and highlighted in the DataTable, and vice versa. This maintains a consistent visual selection state across both views. Figure~\ref{fig:rdf_authoring_panel:linked_selection} illustrates this behavior. 
Algorithms~\ref{alg:rdf_authoring_panel:text_to_rdf_linked_selection} and~\ref{alg:rdf_authoring_panel:rdf_to_text_linked_selection} describe the implementation of linked selection in both directions.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=.95\linewidth]{../figures/MC-RDF-JSON-LD-3.png}
  \caption{Linked selection between Text View and RDF View. The highlighted JSON fragment corresponds to the selected triple in the DataTable, and vice versa.}
  \label{fig:rdf_authoring_panel:linked_selection}
\end{figure}
\label{sec:design:rdf-authoring-panel:view-synchronization:rdf-to-text}

\begin{algorithm}[htbp]
  \caption{Linked selection from Text View to RDF View}
  \label{alg:rdf_authoring_panel:text_to_rdf_linked_selection}
  \begin{algorithmic}[1]
    \State \textbf{Input:} current JSON path selection, full \gls{jsonld} document, current DataTable rows
    \State Read the selected JSON path from the editor state
    \State Extract subject (\texttt{@id}), predicate, and object at the selected path
    \State Build a minimal \gls{jsonld} fragment containing:
    \Statex \hspace{\algorithmicindent}a) original \texttt{@context}
    \Statex \hspace{\algorithmicindent}b) one \texttt{@graph} node with extracted subject, predicate, and object
    \State Parse the minimal fragment with RDFLib into a temporary graph
    \If{the temporary graph contains exactly one statement}
      \State Compare that statement against each DataTable statement by subject, predicate, and object equality
      \If{a matching statement is found}
        \State Determine its row index in the DataTable
        \State Set DataTable selection to the matching row
        \State Scroll/focus the table to the selected row
      \EndIf
    \EndIf
  \end{algorithmic}
\end{algorithm}

\begin{algorithm}[htbp]
  \caption{Linked selection from RDF View to Text View}
  \label{alg:rdf_authoring_panel:rdf_to_text_linked_selection}
  \begin{algorithmic}[1]
    \State \textbf{Input:} selected DataTable statement $(s,p,o)$, current \gls{jsonld} document
    \State Read subject $s$, predicate $p$, and object $o$ from the selected table row
    \State Find candidate node in \texttt{@graph} whose \texttt{@id} matches $s$
    \If{no candidate node is found}
      \State \Return without navigation
    \EndIf
    \State Find predicate entry in the candidate node that matches $p$
    \Statex \hspace{\algorithmicindent}(consider both prefixed terms and full \glspl{iri} as equivalent)
    \If{no matching predicate entry is found}
      \State \Return without navigation
    \EndIf
    \State Match object $o$ within the predicate value
    \Statex \hspace{\algorithmicindent}(as \texttt{@id}, \texttt{@value}, or scalar value)
    \If{an object match is found}
      \State Compute the corresponding JSON path
      \State Move editor cursor/focus to that JSON path
    \EndIf
  \end{algorithmic}
\end{algorithm}

\section{Query with SPARQL}\label{sec:design:query-with-sparql}
The \gls{sparql} query functionality is implemented in \texttt{SparqlEditor.vue} as a dialog with three tabs: \textbf{Query View}, \textbf{Result View}, and \textbf{Visualization}. During implementation, RDFLib query capabilities were evaluated first, but they proved limited for the required in-browser query workflow and did not provide full support for current SPARQL 1.2 draft specifications. Query execution is therefore handled in-browser via Comunica's \texttt{QueryEngine} \cite{comunica_query_sparql}. According to Comunica's official documentation, the engine supports SPARQL 1.2 draft query-related specifications \cite{comunica_supported_specs}.

In the \textbf{Query View}, users can write and validate queries in a CodeMirror editor \cite{codemirror} and execute them against the current in-memory \gls{rdf} graph. Query syntax is checked through debounced live validation using the SparqlJS parser, which provides immediate feedback while editing \cite{sparqljs}. The same view also provides \gls{ai}-assisted \gls{sparql} generation. Similar to the \gls{ai}-assisted \gls{rml} workflow described in Section~\ref{sec:design:rdf-authoring-panel:rml-mapping-tool}, user hints are combined with prompt-engineering instructions and a representative subset of the current \gls{jsonld} data, then sent to the configured \gls{llm} to generate a draft \gls{sparql} query. In simplified form, the prompt includes: (i) output constraints (return only SPARQL), (ii) relevant prefixes/context, (iii) representative graph fragments, and (iv) the user hint. Figure~\ref{fig:rdf_authoring_panel:sparql_ai_generated} shows the dialog with an AI-assisted query suggestion in the Query View.

To demonstrate the \gls{ai}-assisted workflow, we assume that the simulation \gls{json} from Listing~\ref{lst:background_json_example} has been extended with additional \texttt{element\_size} and \texttt{max\_von\_mises\_stress} value pairs (e.g., from multiple simulation runs), and then transformed into \gls{jsonld} using the mapping workflow described earlier. In this extended dataset, metric measurements are available for multiple element sizes, for example \texttt{0.003125}, \texttt{0.00625}, \texttt{0.0125}, \texttt{0.05} and \texttt{0.1}.

In this scenario, the user provides a natural-language hint in the Query View to request a query that returns paired values of \texttt{element\_size} and \texttt{max\_von\_mises\_stress}. An example hint is shown in Listing~\ref{lst:design_sparql_ai_hint}. The hint and a subset of the current \gls{jsonld} graph are sent to the AI API, which proposes a draft query that may still contain syntactic or semantic errors. Therefore, the generated query must be reviewed and, if necessary, refined by the user before execution. A representative generated query is shown in Listing~\ref{lst:design_sparql_ai_generated}.

\begin{lstlisting}[float=htbp,style=plaintext,caption={Example user hint for AI-assisted SPARQL generation}, label={lst:design_sparql_ai_hint}]
Return pairs of element_size and max_von_mises_stress.
For each run, show the numeric element size value and the numeric stress value.
Sort by element_size ascending.
\end{lstlisting}

\begin{lstlisting}[float=htbp,style=sparql,caption={AI-suggested SPARQL query for \texttt{element\_size} vs. \texttt{max\_von\_mises\_stress} pairs}, label={lst:design_sparql_ai_generated}]
PREFIX m4i: <http://w3id.org/nfdi4ing/metadata4ing#>
PREFIX schema: <https://schema.org/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX unit: <http://qudt.org/vocab/unit/>

SELECT ?element_size ?max_von_mises_stress WHERE {
  ?method a m4i:Method .
  ?method m4i:hasParameter ?element_size_param .
  ?element_size_param a schema:PropertyValue ;
                      schema:value ?element_size .
  ?method m4i:investigates ?stress_param .
  ?stress_param a schema:PropertyValue ;
                schema:value ?max_von_mises_stress .
} ORDER BY ASC(?element_size)
\end{lstlisting}

This showcase illustrates the intended interaction pattern: \gls{ai} is used to accelerate query authoring, while final semantic and syntactic validation remains under user control before execution. Human-in-the-loop review is therefore essential in both \gls{ai}-generated \gls{rml} mappings and \gls{ai}-generated \gls{sparql} queries, and users can inspect and refine both results before applying or executing them.

The dialog supports two execution modes controlled by a toggle: \textbf{Visualization Off} and \textbf{Visualization On}. With \textbf{Visualization Off}, standard SELECT-style querying is used and results are shown in the \textbf{Result View}. There, the PrimeVue DataTable columns are generated dynamically from the returned variables, and export is provided as CSV. With \textbf{Visualization On}, the query mode switches to CONSTRUCT-based graph output; the same result table is still populated, but export options target RDF serializations (e.g., Turtle, N-Triples, RDF/XML) for graph-oriented workflows.

The \textbf{Visualization} tab renders the query result graph in read-only mode. Its interaction model and rendering details are discussed in the Section~\ref{sec:design:knowledge-graph-visualization}.

\section{\glsentryshort{kg} Visualization}\label{sec:design:knowledge-graph-visualization}

\gls{kg} rendering is implemented in \texttt{RdfVisualizer.vue} using Cytoscape.js \cite{cytoscapejs}, with the COSE-Bilkent layout extension for force-directed graph placement. The visualization presents resources as nodes and semantic relations as directed edges, while preserving readable labels through namespace-aware prefix shortening.

The view provides interactive graph navigation features including zoom in/out, fit-to-view, and zoom-to-selected-node actions. For performance and usability, large graphs are detected before rendering and users are prompted to continue or cancel. In addition, graph export is supported as an image, and an optional animation mode can recompute the layout to improve readability after edits.

Node inspection and editing support are integrated through \texttt{RdfVisualizerPropertiesView.vue}. When a node is selected, a side panel shows the node identifier and its properties, including linkable IRIs. The same panel exposes lightweight authoring actions (add/edit/delete property, rename/delete node, add node), so users can inspect and refine graph content directly from the visualization workflow.

The visualization tab can be opened in two ways. First, it can be opened directly from the Triples tab of the RDF panel. In this mode, the visualization is editable, and user interactions are propagated back to the main JSON-LD data and RDF store. Second, it can be opened from the SPARQL dialog when the \textbf{Visualization On} toggle is enabled. In that case, the graph is built from the user's SPARQL CONSTRUCT result and is shown in read-only mode, because returned triples are not guaranteed to map one-to-one to statements in the internal RDF store.

For editable visualization workflows, undo and redo actions are also integrated in the graph toolbar. This allows users to safely revise node/property edits while keeping visualization and underlying data synchronized.

To improve navigation in larger graphs, the visualization also provides node search via PrimeVue AutoComplete\footnote{\url{https://primevue.org/autocomplete/}} component: users receive suggestions of available nodes while typing, can quickly select the intended node, and the view automatically focuses on it.

The visualization also supports clickable hyperlinks for semantic terms. Predicate labels can open their ontology definitions; for example, \texttt{m4i:hasParameter} links to predicate definition in the ontology\footnote{\url{https://nfdi4ing.pages.rwth-aachen.de/metadata4ing/metadata4ing/index.html\#hasParameter}}. For object values, \texttt{NamedNode} are handled context-sensitively: if the referenced node exists in the current graph, clicking it focuses that node in the visualization; if it points to an external resource, clicking opens the external definition (e.g., \texttt{unit:M} linking to \url{https://qudt.org/vocab/unit/M}).
